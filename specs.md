# specs.md – Self‑Hosted Valence Listening Test (jsPsych + Google Apps Script)

## 1. Purpose and Scope

This document specifies the design and implementation of a web‑based listening test for collecting **valence ratings** of synthesized musical excerpts.

- **Stimuli:** 30–60 audio excerpts, each 30 seconds long, generated by a separate MIDI → audio pipeline.
- **Participants:** Can be musicians or non‑musicians; only need basic listening ability and access to headphones.
- **Platform:**
  - Front end: Static site (e.g., GitHub Pages) running **jsPsych** with the `audio-slider-response` plugin for trial logic.[web:93][web:125]
  - Back end: **Google Apps Script** web app writing to a private **Google Sheet**; script runs as the researcher and accepts anonymous POSTs from the front end.[web:167][web:170][web:176]
- **Data:** Trial‑level valence ratings + rich behavioral metadata (timestamps, reaction times, interaction traces) stored per participant for later analysis.

This specification should be sufficient for a technically literate researcher to implement the full system and adapt text and parameters as required by ethics approval.

---

## 2. System Architecture

### 2.1 High‑level components

1. **Stimulus generation pipeline (outside this repo)**
   - Inputs: MIDI files and synthesis configuration.
   - Outputs:
     - 30–60 audio files (e.g., `stim_001.wav` … `stim_060.wav`).
     - `stimuli.json` manifest with metadata for each stimulus.

2. **Front end (static site)**
   - Technology: HTML + JavaScript using jsPsych 7+ and `@jspsych/plugin-audio-slider-response`.[web:93][web:125][web:157]
   - Hosting: GitHub Pages or equivalent static hosting.
   - Responsibilities:
     - Load `stimuli.json` and build jsPsych timeline programmatically.
     - Present instructions, practice trials, and main block of trials.
     - Enforce full listening before response.
     - Capture extensive behavioral data (RTs, slider changes, approximate listening times, basic interaction traces).
     - On experiment completion, POST JSON payload to Apps Script endpoint.[web:160][web:161]

3. **Back end (Google Apps Script + Google Sheets)**
   - One Google Spreadsheet with three tabs:
     - `Codes` – valid access codes and status.
     - `ValidData` – rows for participants with recognized access codes.
     - `UnverifiedData` – rows for participants with unrecognized codes.
   - Apps Script web app:
     - `doPost(e)` handler parses JSON from front end.
     - Validates access code against `Codes` sheet.
     - Writes each trial as a row to either `ValidData` or `UnverifiedData`.
     - Optionally marks codes as used (one‑time or limited‑use codes).[web:167][web:170][web:176]

## 3. Data Model

### 3.1 Stimulus manifest (`stimuli.json`)

Generated by the synthesis pipeline and placed in the front‑end directory.

**Location (relative to site root):** `stimuli.json`

**Example schema:**

```json
[
  {
    "id": "stim_001",
    "file": "audio/stim_001.wav",
    "group": "A",
    "notes": "optional free text"
  },
  {
    "id": "stim_002",
    "file": "audio/stim_002.wav",
    "group": "B",
    "notes": "..."
  }
]
```

Required fields:

- `id`: Unique identifier per stimulus (string).
- `file`: Relative URL to the audio file.

Optional fields (for design/analysis):

- `group`: Condition grouping (e.g., model A vs. model B).
- `notes`: Any extra metadata.

### 3.2 Trial‑level data structure (front end)

Each jsPsych trial should at minimum include:

- `participant_id`: Random ID assigned at session start.
- `access_code`: Code typed by participant (verifies recruitment source).
- `stim_id`: Copied from manifest `id`.
- `stim_file`: Copied from manifest `file`.
- `trial_index`: jsPsych trial index.
- `block_type`: e.g., `"practice"` or `"main"`.
- `trial_type`: jsPsych plugin name (`"audio-slider-response"`).
- `response`: Final slider value.
- `rt`: jsPsych reaction time (ms) from trial start to button press.
- `t_trial_start`: Absolute timestamp when trial is created/displayed.
- `t_audio_start`: Absolute timestamp when audio playback starts (approx).
- `t_audio_end`: Absolute timestamp when playback ends.
- `t_first_slider_move`: Absolute timestamp of first slider interaction.
- `slider_events`: Array of `{t, value}` entries for each slider change.
- `mouse_clicks`: Optional array of `{t, x, y, target}` for clicks within trial area.
- Any jsPsych defaults such as `time_elapsed`.[web:161]

These fields can be stored directly on the trial data object in jsPsych; the entire trial object is also serialized as `raw_json` by the Apps Script for later re‑parsing.

### 3.3 Google Sheets columns

Each sheet (`ValidData`, `UnverifiedData`) should have header row:

1. `timestamp` – server time when row written.
2. `access_code` – as sent by participant.
3. `code_valid` – boolean; whether access code matched `Codes` tab.
4. `participant_id` – random ID from jsPsych.
5. `stim_id`
6. `stim_file`
7. `block_type`
8. `trial_index`
9. `trial_type`
10. `response`
11. `rt`
12. `t_trial_start`
13. `t_audio_start`
14. `t_audio_end`
15. `t_first_slider_move`
16. `slider_events_json`
17. `mouse_clicks_json`
18. `raw_json` – full jsPsych trial object as JSON string.

The Apps Script is responsible for mapping incoming JSON onto these columns.

## 4. Participant Flow and Interface Specification

All screens are implemented as jsPsych trials (e.g., `html-button-response` or similar), except the main rating trials which use `audio-slider-response`.[web:93][web:125][web:157]

### 4.1 Access‑code prompt

**Purpose:** Light authentication and recruitment control.

**Behavior:**

- On first load, before consent, display a modal or blocking prompt requesting an **access code**.
- Store result in a global variable `access_code` and add to jsPsych data properties.
- If participant cancels or leaves it blank, proceed but data may be treated as unverified.

**Example UI text:**

> **Access code**  
> Please enter the access code provided to you by the researcher. If you do not have a code, leave this blank and click Continue.

### 4.2 Consent / Information screen

**Plugin:** `html-button-response` with a single “I consent” button.

**Content (to be customized for ethics board):**

- Study purpose (high level: understanding how people perceive emotions in music).
- Duration (e.g., 20–30 minutes depending on excerpt count).
- Procedures: listening to short excerpts and rating how they feel.
- Voluntary participation and right to withdraw at any time.
- Data anonymity and storage (e.g., on encrypted university accounts; no identifying information stored with responses).

**Example button text:**  
`"I have read the information above and consent to participate"`.

### 4.3 General instructions and setup

**Plugin:** `html-button-response`.

**Key points to cover:**

- Use **headphones** where possible.
- Complete the study in a **quiet environment**.
- Adjust system volume to a **comfortable listening level**.
- You will hear **30–60 excerpts**, each ~30 seconds long.
- After each excerpt, you will rate **how pleasant or unpleasant the emotion evoked by the music feels**, using a slider.
- Emphasize that there are **no right or wrong answers**; you should respond based on your own impression.

### 4.4 Musical background questionnaire

**Implementation options:**

- Implement directly as jsPsych trials (e.g., `survey-multi-choice`, `survey-text`, or custom HTML).[web:157]

**Suggested questions:**

1. **Age** (optional): numeric or age bands.
2. **Musical training:**
   - “How many years have you played a musical instrument or sung in a structured setting (e.g., lessons, ensembles)?”  
     - None; 1–2; 3–5; 6–10; 10+.
3. **Formal music education:**
   - “Have you received formal education in music theory, composition, or performance?”  
     - No; Yes, up to high school; Yes, university/college; Yes, postgraduate/professional.
4. **Listening frequency:**
   - “How often do you listen to music?”  
     - Less than once a week; 1–3 days per week; 4–6 days per week; Every day; Several hours per day.
5. **Primary genres (optional multi‑select).**

All questionnaire responses should be stored as part of jsPsych data and included in the payload to Apps Script (either once per participant or repeated per trial as needed).

### 4.5 Explanation of valence (concept screen)

**Goal:** Ensure both musicians and non‑musicians understand what is being rated.

**Content (example, to adapt):**

> **What do we mean by “valence”?**  
>  
> Emotions can feel more or less pleasant. In this study, we focus on **valence**, which is how **pleasant or unpleasant** the emotion feels.  
>  
> - Music with **low valence** tends to feel unpleasant or negative (for example: tense, disturbing, anxious, or very sad in a painful way).  
> - Music with **high valence** tends to feel pleasant or positive (for example: joyful, peaceful, comforting, or uplifting).  
>  
> When you rate each excerpt, please think about **how the music makes you feel emotionally**, not whether you like the piece or think it is good or bad music.  
>  
> There are no right or wrong answers. We are interested in your personal impression.

### 4.6 Practice trial(s)

**Purpose:** Familiarize participants with playback and slider.

**Behavior:**

- Use 1–2 short practice excerpts (can be from outside the main set).
- Same interaction as main trials:
  - Audio must play fully.
  - Slider is disabled during playback.
  - Slider is enabled after playback ends; participant must move it before continuing.
- Provide additional on‑screen reminder if participant tries to continue without moving the slider.

Practice trials should be tagged with `block_type = "practice"` in trial data.

### 4.7 Main rating block

**Number of trials:** 30–60 (configurable).

**Trial type:** `audio-slider-response`.[web:93]

**Core behavior per trial:**

1. Display a blank or minimal screen with instructions and disabled slider.
2. Automatically play audio stimulus once loaded.
3. During playback:
   - Slider is visible but disabled (`response_allowed_while_playing: false`).[web:93]
4. After playback ends:
   - Slider becomes active.
   - Participant moves slider to indicate valence.
   - A “Next” button is enabled once slider has moved at least once (`require_movement: true`).[web:93]
5. On clicking “Next”:
   - Trial ends, trial data (including timestamps and slider trace) are stored.
   - Proceed to next trial.

**Slider spec:**

- Range: `0` (very unpleasant) to `100` (very pleasant) in integer steps.
- Initial position: midpoint `50`.
- Labels: `"Very unpleasant"`, `"Neutral"`, `"Very pleasant"` displayed under the slider.[web:93]

**On‑screen prompt example:**

> Please rate how pleasant or unpleasant the emotion evoked by this music feels to you.

Trials should be presented in randomized order per participant (shuffle `stimuli` array before building the timeline).

### 4.8 Debrief / optional comments

**Content:**

- Short text thanking participant.
- Optional free‑text question (e.g., “Is there anything you would like to share about your experience in this study?”).
- If recruitment platform requires completion code, display it here.

## 5. Behavioral Logging Specification

The following behaviors should be captured on each rating trial:

1. **jsPsych default fields**
   - `rt`: time from trial start to button click (ms).
   - `time_elapsed`: cumulative time since experiment start.
   - `trial_type`, `trial_index`.[web:161]

2. **Custom timestamps**
   - `t_trial_start`: `Date.now()` or `performance.now()` at `on_start`.
   - `t_audio_start`: timestamp when audio actually begins; approximate via `on_load` or `on_play` hook.
   - `t_audio_end`: timestamp when playback ends (via `onended` or plugin event).
   - `t_first_slider_move`: first time slider value changes.

3. **Slider interaction trace**
   - `slider_events`: array of `{t, value}` entries:
     - `t`: timestamp relative to trial start or absolute.
     - `value`: slider numeric value at that change.
   - Implement by attaching an `input` event listener to the slider element in `on_load` and pushing events into an array that is then stored on trial data at `on_finish`.

4. **Mouse click trace (optional)**
   - `mouse_clicks`: array of `{t, x, y, target}` for click events within the experiment container during the trial.
   - Attach a `click` listener in `on_load` and remove it in `on_finish`.

5. **Playback checks**
   - Verify that `t_audio_end - t_audio_start` is approximately 30 s (optionally log discrepancies for debugging).
   - Optionally log a boolean `audio_played_full` if playback reached `ended` state.

This level of logging allows offline reconstruction of participant behavior patterns (e.g., early vs. late slider movement, indecision, inattentive clicking) in addition to the final valence rating.

## 6. Front‑End → Apps Script Communication

### 6.1 Payload format

At experiment end (`on_finish`), front end should collect all jsPsych data and send a JSON payload to the Apps Script URL:[web:160][web:161]

```json
{
  "code": "ACCESS_CODE_ENTERED_BY_PARTICIPANT",
  "participant_id": "RANDOM_ID",
  "trials": [
    {
      "participant_id": "...",
      "access_code": "...",
      "stim_id": "stim_001",
      "stim_file": "audio/stim_001.wav",
      "block_type": "main",
      "trial_index": 5,
      "trial_type": "audio-slider-response",
      "response": 73,
      "rt": 4500,
      "t_trial_start": 12345678,
      "t_audio_start": 12345700,
      "t_audio_end": 12375700,
      "t_first_slider_move": 12376000,
      "slider_events": [
        {"t": 12376000, "value": 30},
        {"t": 12376500, "value": 73}
      ],
      "mouse_clicks": [],
      "time_elapsed": 600000,
      "...": "(any other jsPsych fields)"
    }
  ]
}
```

### 6.2 Front‑end responsibilities

- Store `participant_id` and `access_code` globally and add them via `jsPsych.data.addProperties` at start.
- Ensure all custom fields (timestamps, traces) are attached to trial data before the trial ends.
- Use `fetch(SCRIPT_URL, { method: 'POST', headers: {'Content-Type': 'application/json'}, body: JSON.stringify(payload) })` in `on_finish`.
- Handle success/failure responses by displaying appropriate messages.

## 7. Apps Script / Google Sheets Logic

### 7.1 Codes sheet (`Codes`)

Columns:

1. `code` – access code string (e.g., `A1234`).
2. `label` – optional label (e.g., “Prolific batch 1”).
3. `used` – boolean indicating whether code has been used.

Logic:

- On first valid submission with a given code, mark `used = TRUE` (optional; enables one‑time use semantics).
- Additional submissions with same code can either be allowed but flagged, or rejected based on study needs.

### 7.2 Data handling (`ValidData` / `UnverifiedData`)

Apps Script `doPost(e)` performs:[web:167][web:170][web:176]

1. Parse JSON body.
2. Extract `code`, `participant_id`, and `trials` array.
3. Lookup code in `Codes` sheet:
   - If found: `code_valid = TRUE`, select `ValidData` sheet.
   - If not found: `code_valid = FALSE`, select `UnverifiedData` sheet.
4. For each trial in `trials`, construct a row matching the column order in Section 3.3.
5. Append all rows to chosen sheet in a single `setValues` call.
6. Return JSON response `{status: 'ok', valid: true/false}`.

This separation allows you to manually inspect unverified rows and decide whether to include them in analysis.

## 8. Deployment and Maintenance

### 8.1 Front‑end hosting

- Host `index.html`, `experiment.js`, `stimuli.json`, and `audio/` folder on GitHub Pages or equivalent static hosting.[web:144][web:147]
- Ensure CORS is not an issue when calling the Apps Script URL (Apps Script web apps support cross‑origin POSTs by default when configured as public).[web:170][web:184][web:193]

### 8.2 Updating stimuli

When adding or changing stimuli:

1. Run synthesis pipeline to generate new audio files into `audio/`.
2. Regenerate `stimuli.json` using the pipeline script.
3. Commit and push changes to GitHub; GitHub Pages redeploys automatically.[web:144][web:147]

### 8.3 Monitoring and debugging

- Use browser dev tools console to verify:
  - `stimuli.json` loads correctly.
  - Audio files return 200 OK.
  - `fetch` to Apps Script returns 200 and `{status: 'ok'}`.[web:160][web:161]
- Use Google Sheets to inspect recent rows in `ValidData` and `UnverifiedData`.
- Use Apps Script logs to debug JSON parsing or range issues.

---

## 9. Ethical and Practical Considerations

- Ensure language in consent and debrief matches IRB/ethics requirements (data retention, anonymization, withdrawal procedures).
- If collecting IPs or any identifying metadata (not recommended), this must be disclosed and minimized.
- Consider including a headphone screening or attention checks if data quality is a concern.
- Clearly state approximate study length to minimize mid‑study attrition.

---

This specification intentionally focuses on clear division of responsibilities between interface, data logging, and storage, so that each can be implemented and tested independently while preserving the ability to collect rich behavioral traces alongside valence ratings.
